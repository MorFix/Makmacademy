Git (Extensions) Introduction

Git is the most common source control tool. Git is a bash command line and can be used with Git Extensions which is a UI interface for running git commands. In Git Extensions most of the commands are available with right click, the toolbar buttons or the menu.

  • A git practical tutorial is online at This path

Commit

To pack several changes together, you commit them. This is done by selecting changed files (or changed lines in files) and staging them.

Push

To send the changes to the server for others to pull you simply push them to a server.

Your git server can be either a repository in TFS (or GitHub outside the army) or even a simple shared directory.

Branch

Git is based on branches. Although you can work on a single branch, it is recommended to create a new one when starting a new task and merging it at the end of the task. It's also a good idea to have a development branch which will be merged to a master branch every time you want to produce a new version of your software.

Fetch/Pull

To get changes from the server you can pull or fetch. Fetching means that you only get the information about the remote branches and don't change anything in your local ones. If you pull you can select what will happen to your branch if it has changes (if it will be merged / rebased / fast forwarded ..)

Merge

You can merge one branch into another to pass all commits combined to it.

Rebase

You can change the history of a branch by telling git to take all of the commits it has, make the branch start from a different commit in the head of another branch and replaying all the commits from there.

.gitignore

There is a need to ignore certain files or file types when committing and pushing changes.

Each line in a .gitignore file specifies a pattern of files or file types to ignore.

GitExtensions offer you default gitignore patterns. You can use them, but you must double check and customize it for your needs.

For example, you must not commit the node_modules folder when developing a Web application, and clearly it differs from the suggested default patterns.

But instead filter those file types that are relevant to your project, and\or add some of your own.
